# ProScan Feature Catalog — Deduped, Scored, Tiered

_Last updated: 2026-06-10 — produced by the planning fleet; prerequisites amended after completeness review._

This catalog consolidates 58 raw feature ideas from four persona-ideation passes (First-Flip Felix, Storefront-Stalker Sarah, Operator Omar, Blue-sky/AI) into 34 surviving features across four tiers, plus merged/rejected notes. It is grounded in the extension audit, competitor landscape, seller-workflow, technical, and monetization research in `docs/research/`, and in `REVAMP_PLAN.md` (phases referenced below are that plan's phases).

## Scoring method

Each surviving feature is scored 1–5 on three axes:

- **SV — Seller value.** Does it replace the Excel workflow and/or drive retention? 5 = the seller would notice its absence weekly; 1 = nice-to-have.
- **DF — Differentiation.** Position vs Keepa (https://keepa.com), SellerAmp (https://selleramp.com), Seller Assistant (https://sellerassistant.app), Tactical Arbitrage (https://tacticalarbitrage.com), Helium 10 (https://helium10.com), SmartScout (https://smartscout.com). 5 = no surveyed tool has it; 3 = exists elsewhere but gated/worse; 1 = commodity.
- **FS — Feasibility.** Within Firebase (Spark-first) + MV3 constraints. Dashboard-only work scores higher than extension changes; anything needing Cloud Functions/Blaze, new scraping surfaces, or a security-rules redesign scores lower. 5 = pure dashboard over existing data; 1 = new architecture.

Scores are shown as **SV/DF/FS**. Effort is S/M/L/XL relative to one engineering session-week. Tier rules applied:

- **MVP** = the smallest platform that already beats the Excel workbook: auth + sync + core tracking. No Cloud Functions except the one unavoidable callable (see prerequisites). Owner's explicit asks (month-over-month price/rating/review tracking; spread/variation analysis; what-to-scrape/what-to-flip) land in MVP or v1.
- **v1** = fast-follows within weeks of MVP, mostly dashboard work over data MVP already syncs.
- **v2** = the differentiation wave; includes everything requiring Blaze (Functions/Scheduler/email).
- **later** = parked: real value but heavy (multi-user, server-side LLM) or dependent on accumulated history.
- **rejected** = bad fit, with reasons.

## Sequencing notes for engineering

1. **Extension changes cluster into three batches**, each independently shippable to the Chrome Web Store: (a) MVP prerequisites + sync (scrapeRun, numeric price, dedup, sponsored flag, spread fixes, key removal, dead-code deletion, auth handoff, queue writer); (b) the v1 AOD parse (seller IDs, offer count, FBA/FBM, Buy Box) + two SERP selectors (demand, image) + ARM_SCRAPE/maxPages; (c) nothing — v2 and Later are dashboard/Functions work except trivial counters. Batch CWS submissions accordingly; review times are unpredictable.
2. **The BSR/category PDP fetch is deliberately absent from every tier above.** It is the biggest data gap (sellers' #1 threshold input) but it adds a new scraping surface with rate-limit and Amazon Agent Policy exposure; revisit after v1 ships and the offer-fetcher's jittered pattern has proven uneventful in production.
3. **Monetization hooks land with v2**: watchlist count and history depth are the gates (per the monetization research's Ladder A: free forever + Pro $14.99/mo), and the digest is the conversion surface. Nothing in MVP/v1 should be paywalled — current extension features stay free.
4. The 214-test Jest fixture suite is the contract for every extension change above; extend fixtures for each new selector before writing the selector. The selector + AOD verification spike (see prerequisites) is the mechanism: it front-loads live-page verification and fresh fixture capture for batch (a) and the AOD cadence test for batch (b), and runs before either batch starts.
